# TB ŌĆö GPS Map Data Pipeline

## Overview

The GPS map ([FB-gps-map](../feature-briefs/FB-gps-map.md)) is rendered from a single SVG file generated by a Python pipeline. This brief specifies the pipeline: the inputs, the intermediate annotations, the SVG output format (including localisation-tag conventions), and the UE-side ingest path.

The same SVG drives the in-game GPS, an annotated-development PNG, and external rendering for marketing or technical docs. Generating once and rendering many times keeps the source of truth singular.

**Related documents:**

- [FB-gps-map](../feature-briefs/FB-gps-map.md) ŌĆö player-facing design that this pipeline supports
- [UI/UX](../gdd/ui-ux.md) ŌĆö phone UI host
- [Terridyn City Grid](../../mgaPriv/world/terridyn-city-grid.md) ŌĆö block addressing, river-block table, Central Park bounds (private)
- [reference/organize_roads.py](../../reference/organize_roads.py) ŌĆö produces `roads_organized.json` (intersection lookup)
- [reference/road_map_check.py](../../reference/road_map_check.py) ŌĆö Stage 1 (geometry truth from road-height PNG)

## Pipeline Architecture

Three stages, each writing a structured JSON intermediate:

```
junctions.json          ŌöĆŌöÉ
roads_organized.json    ŌöĆŌöż
road-height PNG export  ŌöĆŌö┤ŌöĆŌåÆ  road_map_check.py       ŌöĆŌöĆŌåÆ  block_status.json   (Stage 1)
                                                          Ōåō
districts.json (areas) + ŌöĆŌåÆ  district_assign.py        ŌöĆŌöĆŌåÆ  block_annotations.json   (Stage 2)
  generated schools/camps     Ōåō
                              merges road status + districts + schools
                                                          Ōåō
block_annotations.json   ŌöĆŌåÆ  gps_map_render.py         ŌöĆŌöĆŌåÆ  downtown.svg   (Stage 3)
                                                              Ōö£ŌöĆŌåÆ  in-game asset (UE import)
                                                              Ōö£ŌöĆŌåÆ  annotated dev PNG (rasterise)
                                                              ŌööŌöĆŌåÆ  marketing PNG (alt styling)
```

Each stage is independently re-runnable. A change to the road network re-runs Stage 1; a change to district assignments re-runs Stage 2; a styling change re-runs Stage 3 only.

## Stage 2 ŌĆö District annotation (`district_assign.py`)

**Built (2026-06-10).** `reference/district_assign.py` reads the hand-authored
area placements + the generated school/campus placements and writes
`exports/city_grid/block_annotations.json`.

**Input format (as built):**
- **`districts.json`** ŌĆö the hand-authored source of truth for every named
  district / corp: a list of areas, each with `key, label, owner, land_use,
  tags, color` and an explicit `blocks` list of `db` addresses. Chosen over the
  painted-PNG approach below because corp/landmark footprints are *precise* db
  block lists, not fuzzy regions ŌĆö explicit JSON is exact and diff-friendly.
  A `generated` block records the school/campus generator params.
- **Generated overlays** ŌĆö primary schools (2├Ś2 = 3-school-block L + park) and
  combined secondary campuses (12-block L per superblock) are placed
  procedurally by the generators in `gps_map_render.py` (even lattice, within
  one superblock, on buildable land, off-water), and stamped into the output.

**Future input option (deferred):** a **coarse painted-PNG overlay + CSV legend**
remains the right tool for the *broad residential / mixed-use fill* (fuzzy
zones over hundreds of blocks) once that phase begins ŌĆö paint-bucket authoring,
one colour per land-use band. It would layer *under* the explicit `districts.json`
(explicit wins). Not needed for the precise corp/education placement done so far.

## Stage 3 ŌĆö SVG render (`gps_map_render.py`)

**First cut implemented** (`reference/gps_map_render.py`) ŌĆö renders the layers we already have, ahead of Stage 2:

- **Roads** from `junctions.json` segments, by tier (L1/L2/L3), drawn to each segment's `realX2/realY2` (the true junction) rather than `x2/y2` ŌĆö L2/L3 carry a 100 m `END_SETBACK` that leaves a deliberate gap at junctions for the UE spline tool, and drawing the truncated end leaves the arterials disconnected (L-shaped doglegs). Each segment is then **clipped to the built area** rather than kept-or-dropped whole (`built_intervals()`): walk the `RT_RoadHeights` texture along the centreline (~6 m spacing), test each sample for road within ┬▒40 m perpendicular, bridge short interior gaps (Ōēż30 m: intersections / RT artefacts), split at long gaps, and emit each surviving run (Ōēź28 m) as its own polyline + label. River and frayed-edge handling is **inherent** ŌĆö the road simply stops at the water or the built boundary. Bridges are NOT in `junctions.json` ŌĆö they belong in the deferred `layer-bridges`. `--no-cull` draws the full designed grid.

  Two corrections matter for the clip to find the right roads: **(a) calibrated transform** ŌĆö the nominal RT worldŌåÆtexel scale carries a documented ~1.6 %-class stretch/offset that grows toward the landscape edges (Ōēł50 m at the downtown's north edge), so `calibrate_rt()` fits an affine from the actual RT intersection centroids (sub-pixel residual) instead of trusting the nominal scale; **(b) the ┬▒40 m perpendicular search** ŌĆö the procedural grid's internal L1 roads sit ~15ŌĆō50 m off the as-built RT, so without it real-but-offset roads vanish.

  **Why clipping, not whole-segment keep/cull:** a `junctions.json` segment spans a full superblock width, but a partial/irregular superblock may build only part of that span (the rest is unbuilt land). A whole-segment test rejects the entire road because of the long unbuilt run ŌĆö symptom: roads bordering valid-but-partial-superblock blocks (e.g. db17ŌĆō19 / rows BŌĆōE) disappear even though the RT clearly has road there. Clipping keeps the built portion and drops the unbuilt extension. (The earlier whole-segment longest-black-run test is retained as `segment_built()` but no longer used by the renderer.)

  NOTE: `road_map_check_rt.py` (Stage 1) still uses the nominal transform, so `block_availability.csv` carries the same uncorrected offset near the edges ŌĆö worth porting `calibrate_rt` there too.
- **Block road-status** fills from `block_availability.csv` (`block-present` / `block-partial` / `block-missing` classes). Blocks that sit **on the river** (centre or Ōēź2 corners inside the survey water polygon) are dropped ŌĆö the water layer represents them ŌĆö so no block fill crosses the river. Dry removed voids (e.g. the rectangular hole) are retained.
- **River** water polygon built by offsetting the survey centreline by its per-point width (`river_survey_ŌĆ”_splines_ŌĆ”.json`).
- **Lock markers** at the active pool transitions (Lock 6 removed ŌåÆ 5 markers).
- Writes `exports/city_grid/downtown_blocks.svg`; `--preview <png>` also rasterises via PIL (no system SVG rasteriser required).

- **Block-address labels** ŌĆö literal `dbŌĆ”` text (yellow, `.block-label` class) at each drawn block's centroid in `layer-blocks-text`. Per the TB these are NOT localised. On-water blocks (dropped) get no label.
- **Road-name labels** ŌĆö one black label per built road sub-segment in `layer-streets`, centred on the (clipped) road piece and rotated along it. Font size is **per-tier = `--road-label-frac` ├Ś the tier's carriageway width** (default 0.9 ŌåÆ L1 16.2, L2 27, L3 36 m), so the label height tracks the road width (`MERIDIAN`/`MAIN` large on L3, L1 streets small). The name is resolved by nearest-intersection lookup against `roads_organized.json` (matching-orientation short name: `23N`, `11W`, `MERIDIAN`, `MAIN`), carried as visible text plus `data-name-short`/`data-tier`. Style (fill/size/anchor/baseline) is **inlined per element**, not via a CSS class ŌĆö and crucially NOT via the tier *path* class (`road-l3` etc.), which carries `stroke-width`/`fill:none` for the road LINE and would paint a thick stroke over the glyphs. This is the **dev/annotation styling** (human-readable names so block + road call-outs are easy in Illustrator); the in-game variant should swap the visible text for `{LOC:gps.street.*}` tags (see Output Variants) ŌĆö still TODO.

Emits the TB layer-group structure, `id`/`class`/`data-*` conventions below. **Still deferred:** the district layer (`layer-districts` empty until Stage 2), `{LOC:gps.street.*}` tag mode for the in-game street labels, bridges, and runtime-overlay groups (left empty for UE injection).

Full specification follows:

**Output:** a single SVG file (`downtown_blocks.svg`) covering the L3-bounded downtown with all visual layers from [FB-gps-map ┬¦ Visual Layers](../feature-briefs/FB-gps-map.md#visual-layers).

**SVG structure (top-level groups, ordered back-to-front):**

```xml
<svg viewBox="..." xmlns="...">
  <defs>
    <!-- shared symbols, patterns, gradients -->
  </defs>

  <g id="layer-water">      <!-- river polygon, pool fills -->         </g>
  <g id="layer-districts">  <!-- per-block fill rectangles, fill colour by district -->  </g>
  <g id="layer-blocks">     <!-- block-boundary outlines -->           </g>
  <g id="layer-roads-l1">   <!-- L1 polylines -->                      </g>
  <g id="layer-roads-l2">   <!-- L2 polylines -->                      </g>
  <g id="layer-roads-l3">   <!-- L3 polylines -->                      </g>
  <g id="layer-bridges">    <!-- bridge geometry -->                   </g>
  <g id="layer-landmarks">  <!-- labelled landmark shapes -->          </g>
  <g id="layer-streets">    <!-- street-name <text> along centrelines  -->  </g>
  <g id="layer-blocks-text"><!-- block address <text> at block centres -->  </g>

  <!-- runtime-overlay groups left empty; UE injects content -->
  <g id="layer-player"></g>
  <g id="layer-pins"></g>
  <g id="layer-pois-discovered"></g>
  <g id="layer-private-overlays"></g>
</svg>
```

**ID conventions:**
- Each block element: `id="block-db1A"` ŌĆö UE can query/style any block by db address
- Each road: `id="road-12W"`, `id="road-main"`, `id="road-3rdN"` ŌĆö UE can query by street short name
- Each district element: `id="district-central-park"`
- Each landmark: `id="landmark-lock-4"`

**Style classes** for tier-based and use-based styling (UE can override at runtime):
- Roads: `road-l1`, `road-l2`, `road-l3`, `road-bridge`
- Districts (by land use): `district-residential`, `district-commercial`, `district-industrial`, etc.
- Blocks (by status): `block-present`, `block-partial`, `block-missing`, `block-private`

CSS embedded in `<defs>` provides default styling; UE can append a stylesheet that overrides for production look (different colour palette, different stroke weights, etc.).

## Localisation Tags

All player-visible text in the SVG is **emitted as a localisation tag, not a literal string.** Tag format:

```
{LOC:<namespace>.<key>}
```

Examples:
- `<text id="label-road-12W">{LOC:gps.street.12w}</text>`
- `<text id="label-landmark-central-park">{LOC:gps.landmark.central_park}</text>`
- `<text id="label-district-shipwright-mall">{LOC:gps.district.shipwright_mall}</text>`
- `<text id="label-block-db1A">db1A</text>`  ŌåÉ block address, NOT localised

**Namespaces** (under `gps.*`):
- `gps.street.*` ŌĆö Streets, Avenues, Meridian, Main
- `gps.landmark.*` ŌĆö Named landmarks (parks, locks, towers, the docks)
- `gps.district.*` ŌĆö District names (malls, strips, named neighbourhoods)
- `gps.poi.*` ŌĆö Discovered POIs (taverns, shops, etc.)
- `gps.label.*` ŌĆö UI labels (compass, search placeholder, "you are here")

**Key naming rules:**
- Lower-case, snake_case
- Numeric streets: `gps.street.12w`, `gps.street.3rd_n`, `gps.street.meridian`, `gps.street.main`
- Landmarks: `gps.landmark.central_park`, `gps.landmark.lock_4`, `gps.landmark.gov_tower`
- District names from the canonical lore document, snake-cased

The Python renderer must NOT emit raw English text for these categories ŌĆö emitting a tag is mandatory. A failed lookup at runtime should display the tag itself (`{LOC:gps.street.12w}`) so missing translations are visually obvious during testing.

### Harvesting tags into the string table

UE imports the SVG. At import time, a harvester pass scans the SVG for `{LOC:*}` markers, extracts unique tag keys, and ensures each one exists in the GPS string table. New keys without translations are added with their English value defaulted from the canonical lore-name (passed through alongside the SVG as a `gps_strings_default.csv`).

Workflow:
1. Python renderer emits `downtown_blocks.svg` (tags only) and `gps_strings_default.csv` (key ŌåÆ English default text).
2. Engine import script merges defaults into `StringTable_GPS.csv`, preserving existing translations, adding new keys with English fallback.
3. Localisation pass on the string table produces locale variants ŌĆö these never round-trip back through the Python pipeline.

## Per-Annotation Custom Data on SVG Elements

Beyond localisation tags, useful per-block / per-road metadata is encoded as `data-*` attributes ŌĆö readable by UE at import time for runtime queries (e.g., "what district owns this block?"):

```xml
<rect id="block-db1A"
      class="block-present district-residential"
      x="..." y="..." width="..." height="..."
      data-db="db1A"
      data-sb-col="-6" data-sb-row="5"
      data-district="ROH housing"
      data-owner="Mautorium"
      data-land-use="residential"
      data-road-status="present" />

<text id="label-road-12W"
      class="road-label road-l3"
      x="..." y="..."
      transform="rotate(...)"
      data-tier="L3" data-direction="ns"
      data-name-short="12W">{LOC:gps.street.12w}</text>
```

The UE import process can read these attributes into structured metadata stored alongside the texture.

## UE-Side Integration

**Import path:**
1. SVG file dropped into `Content/MGA/Maps/`
2. Custom asset importer runs the localisation harvester, populates `StringTable_GPS`
3. SVG is converted to a UE-renderable format (SVG-to-vector-texture via plugin, or rasterised to multi-resolution tile set, depending on engine support)
4. `data-*` metadata extracted into a `MapData_Downtown` structured asset (block-address-keyed dictionary, road-name-keyed dictionary)

**Runtime:**
1. The phone's Map widget loads the map texture / vector source
2. Player marker, pins, discovered POIs rendered as overlay groups (the empty `<g>` runtime layers)
3. Zoom and pan via standard UMG / SlateBrush logic
4. Search hits the `MapData_Downtown` dictionary for block / street / landmark lookups; localised display text via string table

**Decisions to make during implementation:**
- SVG ŌåÆ vector texture vs SVG ŌåÆ tiled raster. Both are viable; tile rasterisation is simpler but loses true vector zoom. A vector approach (e.g. Lottie, NanoSVG, or a custom rasteriser at zoom-time) preserves crispness at any zoom but adds engine-side complexity.
- Whether `data-*` extraction happens at import or runtime. Import-time is faster; runtime allows live re-import without engine restart.

## Update Workflow

Typical iteration:

| Event | Re-run | Why |
|-------|--------|-----|
| New road-height export (more roads laid down) | Stage 1 ŌåÆ 2 ŌåÆ 3 | Block status changed |
| Updated district painting | Stage 2 ŌåÆ 3 | District assignments changed |
| Renamed a landmark | Stage 3 only (and string table) | Just a string change, optionally a tag rename |
| Changed SVG visual style | Stage 3 only | Renderer config |
| Added a new POI category | Stage 3 + UE import | New layer / new class |

`organize_roads.py` (which produces `roads_organized.json` and `roads_organized.txt`) does NOT need to re-run between iterations ŌĆö its inputs are the procedural grid, which is stable. It re-runs only if the grid spec changes (e.g. an L3 boundary moves).
