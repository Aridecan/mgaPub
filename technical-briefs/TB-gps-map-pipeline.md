# TB — GPS Map Data Pipeline

## Overview

The GPS map ([FB-gps-map](../feature-briefs/FB-gps-map.md)) is rendered from a single SVG file generated by a Python pipeline. This brief specifies the pipeline: the inputs, the intermediate annotations, the SVG output format (including localisation-tag conventions), and the UE-side ingest path.

The same SVG drives the in-game GPS, an annotated-development PNG, and external rendering for marketing or technical docs. Generating once and rendering many times keeps the source of truth singular.

**Related documents:**

- [FB-gps-map](../feature-briefs/FB-gps-map.md) — player-facing design that this pipeline supports
- [UI/UX](../gdd/ui-ux.md) — phone UI host
- [Terridyn City Grid](../../mgaPriv/world/terridyn-city-grid.md) — block addressing, river-block table, Central Park bounds (private)
- [reference/organize_roads.py](../../reference/organize_roads.py) — produces `roads_organized.json` (intersection lookup)
- [reference/road_map_check.py](../../reference/road_map_check.py) — Stage 1 (geometry truth from road-height PNG)

## Pipeline Architecture

Three stages, each writing a structured JSON intermediate:

```
junctions.json          ─┐
roads_organized.json    ─┤
road-height PNG export  ─┴─→  road_map_check.py       ──→  block_status.json   (Stage 1)
                                                          ↓
districts_input (PNG +   ─→  district_assign.py        ──→  block_annotations.json   (Stage 2)
  CSV legend, or CSV)         ↓
                              merges status + districts
                                                          ↓
block_annotations.json   ─→  gps_map_render.py         ──→  downtown.svg   (Stage 3)
                                                              ├─→  in-game asset (UE import)
                                                              ├─→  annotated dev PNG (rasterise)
                                                              └─→  marketing PNG (alt styling)
```

Each stage is independently re-runnable. A change to the road network re-runs Stage 1; a change to district assignments re-runs Stage 2; a styling change re-runs Stage 3 only.

## Stage 2 — District annotation (`district_assign.py`)

**Not yet built.** Specification:

**Input format (recommended hybrid):**
- **Coarse PNG overlay** — a painted image at the same dimensions and orientation as the road-height export, where each district is a flat colour region. Authoring is fast (paint-bucket in any image editor).
- **CSV legend** — `colour_hex,district_name,owner,land_use,tags` mapping each colour in the PNG to its district metadata.
- **Per-block override CSV** — optional second CSV `db,district_name,owner,land_use,notes` for blocks whose assignment differs from the painted colour (typically named landmarks: Central Park parking, Lock 4 footprint, the Liza Shipwright Mall, etc.). Override CSV wins over PNG-derived assignment.

**Schema:** each block carries two orthogonal dimensions plus optional tags:

```json
{
  "db1A": {
    "road": "present",
    "district": "ROH housing — Mautorium",
    "owner": "Mautorium",
    "land_use": "residential",
    "tags": []
  },
  "db20AF": {
    "road": "present",
    "district": "Central Park — Parking",
    "owner": "City of Terridyn",
    "land_use": "infrastructure",
    "tags": ["parking", "non-buildable"]
  },
  "db17AI": {
    "road": "missing",
    "district": "Lock 4",
    "owner": "TMC",
    "land_use": "infrastructure",
    "tags": ["river", "lock"]
  }
}
```

**`land_use` controlled vocabulary** (initial):
`residential`, `commercial`, `industrial`, `civic`, `parkland`, `infrastructure`, `mixed`, `entertainment`, `corporate`, `educational`, `unassigned`.

**`owner`** is free-form for now — corporation names, "City of Terridyn", "Public", gang names, "None". Will eventually be constrained against the corp file list.

**`tags`** array — auxiliary flags. Examples: `parking`, `non-buildable`, `river`, `lock`, `bridge`, `landmark`, `secret` (acquired-only on the GPS).

## Stage 3 — SVG render (`gps_map_render.py`)

**First cut implemented** (`reference/gps_map_render.py`) — renders the layers we already have, ahead of Stage 2:

- **Roads** from `junctions.json` segments, by tier (L1/L2/L3), drawn to each segment's `realX2/realY2` (the true junction) rather than `x2/y2` — L2/L3 carry a 100 m `END_SETBACK` that leaves a deliberate gap at junctions for the UE spline tool, and drawing the truncated end leaves the arterials disconnected (L-shaped doglegs). Roads are **culled to the built area** by walking the `RT_RoadHeights` texture along each segment centreline (~6 m spacing, same verified transform as Stage 1) and rejecting any segment whose longest continuous non-road gap exceeds ~30 m. A present road is white along its whole length; a river/void crossing has a long black run, so this culls river-crossing grid segments cleanly (validated: 0 of 259 river-crossers kept, vs 56 under a naive fraction test). Bridges are NOT in `junctions.json` — they belong in the deferred `layer-bridges`. `--no-cull` draws the full designed grid.
- **Block road-status** fills from `block_availability.csv` (`block-present` / `block-partial` / `block-missing` classes). Blocks that sit **on the river** (centre or ≥2 corners inside the survey water polygon) are dropped — the water layer represents them — so no block fill crosses the river. Dry removed voids (e.g. the rectangular hole) are retained.
- **River** water polygon built by offsetting the survey centreline by its per-point width (`river_survey_…_splines_….json`).
- **Lock markers** at the active pool transitions (Lock 6 removed → 5 markers).
- Writes `exports/city_grid/downtown_blocks.svg`; `--preview <png>` also rasterises via PIL (no system SVG rasteriser required).

Emits the TB layer-group structure, `id`/`class`/`data-*` conventions below. **Still deferred:** the district layer (`layer-districts` empty until Stage 2), street-name and block-address text layers (need `{LOC:gps.*}` tags), bridges, and runtime-overlay groups (left empty for UE injection).

Full specification follows:

**Output:** a single SVG file (`downtown_blocks.svg`) covering the L3-bounded downtown with all visual layers from [FB-gps-map § Visual Layers](../feature-briefs/FB-gps-map.md#visual-layers).

**SVG structure (top-level groups, ordered back-to-front):**

```xml
<svg viewBox="..." xmlns="...">
  <defs>
    <!-- shared symbols, patterns, gradients -->
  </defs>

  <g id="layer-water">      <!-- river polygon, pool fills -->         </g>
  <g id="layer-districts">  <!-- per-block fill rectangles, fill colour by district -->  </g>
  <g id="layer-blocks">     <!-- block-boundary outlines -->           </g>
  <g id="layer-roads-l1">   <!-- L1 polylines -->                      </g>
  <g id="layer-roads-l2">   <!-- L2 polylines -->                      </g>
  <g id="layer-roads-l3">   <!-- L3 polylines -->                      </g>
  <g id="layer-bridges">    <!-- bridge geometry -->                   </g>
  <g id="layer-landmarks">  <!-- labelled landmark shapes -->          </g>
  <g id="layer-streets">    <!-- street-name <text> along centrelines  -->  </g>
  <g id="layer-blocks-text"><!-- block address <text> at block centres -->  </g>

  <!-- runtime-overlay groups left empty; UE injects content -->
  <g id="layer-player"></g>
  <g id="layer-pins"></g>
  <g id="layer-pois-discovered"></g>
  <g id="layer-private-overlays"></g>
</svg>
```

**ID conventions:**
- Each block element: `id="block-db1A"` — UE can query/style any block by db address
- Each road: `id="road-12W"`, `id="road-main"`, `id="road-3rdN"` — UE can query by street short name
- Each district element: `id="district-central-park"`
- Each landmark: `id="landmark-lock-4"`

**Style classes** for tier-based and use-based styling (UE can override at runtime):
- Roads: `road-l1`, `road-l2`, `road-l3`, `road-bridge`
- Districts (by land use): `district-residential`, `district-commercial`, `district-industrial`, etc.
- Blocks (by status): `block-present`, `block-partial`, `block-missing`, `block-private`

CSS embedded in `<defs>` provides default styling; UE can append a stylesheet that overrides for production look (different colour palette, different stroke weights, etc.).

## Localisation Tags

All player-visible text in the SVG is **emitted as a localisation tag, not a literal string.** Tag format:

```
{LOC:<namespace>.<key>}
```

Examples:
- `<text id="label-road-12W">{LOC:gps.street.12w}</text>`
- `<text id="label-landmark-central-park">{LOC:gps.landmark.central_park}</text>`
- `<text id="label-district-shipwright-mall">{LOC:gps.district.shipwright_mall}</text>`
- `<text id="label-block-db1A">db1A</text>`  ← block address, NOT localised

**Namespaces** (under `gps.*`):
- `gps.street.*` — Streets, Avenues, Meridian, Main
- `gps.landmark.*` — Named landmarks (parks, locks, towers, the docks)
- `gps.district.*` — District names (malls, strips, named neighbourhoods)
- `gps.poi.*` — Discovered POIs (taverns, shops, etc.)
- `gps.label.*` — UI labels (compass, search placeholder, "you are here")

**Key naming rules:**
- Lower-case, snake_case
- Numeric streets: `gps.street.12w`, `gps.street.3rd_n`, `gps.street.meridian`, `gps.street.main`
- Landmarks: `gps.landmark.central_park`, `gps.landmark.lock_4`, `gps.landmark.gov_tower`
- District names from the canonical lore document, snake-cased

The Python renderer must NOT emit raw English text for these categories — emitting a tag is mandatory. A failed lookup at runtime should display the tag itself (`{LOC:gps.street.12w}`) so missing translations are visually obvious during testing.

### Harvesting tags into the string table

UE imports the SVG. At import time, a harvester pass scans the SVG for `{LOC:*}` markers, extracts unique tag keys, and ensures each one exists in the GPS string table. New keys without translations are added with their English value defaulted from the canonical lore-name (passed through alongside the SVG as a `gps_strings_default.csv`).

Workflow:
1. Python renderer emits `downtown_blocks.svg` (tags only) and `gps_strings_default.csv` (key → English default text).
2. Engine import script merges defaults into `StringTable_GPS.csv`, preserving existing translations, adding new keys with English fallback.
3. Localisation pass on the string table produces locale variants — these never round-trip back through the Python pipeline.

## Per-Annotation Custom Data on SVG Elements

Beyond localisation tags, useful per-block / per-road metadata is encoded as `data-*` attributes — readable by UE at import time for runtime queries (e.g., "what district owns this block?"):

```xml
<rect id="block-db1A"
      class="block-present district-residential"
      x="..." y="..." width="..." height="..."
      data-db="db1A"
      data-sb-col="-6" data-sb-row="5"
      data-district="ROH housing"
      data-owner="Mautorium"
      data-land-use="residential"
      data-road-status="present" />

<text id="label-road-12W"
      class="road-label road-l3"
      x="..." y="..."
      transform="rotate(...)"
      data-tier="L3" data-direction="ns"
      data-name-short="12W">{LOC:gps.street.12w}</text>
```

The UE import process can read these attributes into structured metadata stored alongside the texture.

## Output Variants

| Variant | Renderer config | Output |
|---------|-----------------|--------|
| **In-game SVG** | Default styling, all localisation tags | `downtown_blocks.svg` — imported into UE as map data |
| **Annotated dev PNG** | `--dev` flag: block labels at ALL zooms, English default labels (tag substituted from default CSV), white background, technical grid optional | `downtown_annotated.png` (rasterised at e.g. 4096²) |
| **Annotated dev SVG** | Same as dev PNG but vector output | `downtown_annotated.svg` — for printing or large viewing |
| **Marketing PNG** | `--style marketing` — boosted colours, optional atmosphere effect, transparent background | `downtown_marketing.png` |

The renderer's modes share the same data; only style configuration and label-suppression rules differ.

## UE-Side Integration

**Import path:**
1. SVG file dropped into `Content/MGA/Maps/`
2. Custom asset importer runs the localisation harvester, populates `StringTable_GPS`
3. SVG is converted to a UE-renderable format (SVG-to-vector-texture via plugin, or rasterised to multi-resolution tile set, depending on engine support)
4. `data-*` metadata extracted into a `MapData_Downtown` structured asset (block-address-keyed dictionary, road-name-keyed dictionary)

**Runtime:**
1. The phone's Map widget loads the map texture / vector source
2. Player marker, pins, discovered POIs rendered as overlay groups (the empty `<g>` runtime layers)
3. Zoom and pan via standard UMG / SlateBrush logic
4. Search hits the `MapData_Downtown` dictionary for block / street / landmark lookups; localised display text via string table

**Decisions to make during implementation:**
- SVG → vector texture vs SVG → tiled raster. Both are viable; tile rasterisation is simpler but loses true vector zoom. A vector approach (e.g. Lottie, NanoSVG, or a custom rasteriser at zoom-time) preserves crispness at any zoom but adds engine-side complexity.
- Whether `data-*` extraction happens at import or runtime. Import-time is faster; runtime allows live re-import without engine restart.

## Update Workflow

Typical iteration:

| Event | Re-run | Why |
|-------|--------|-----|
| New road-height export (more roads laid down) | Stage 1 → 2 → 3 | Block status changed |
| Updated district painting | Stage 2 → 3 | District assignments changed |
| Renamed a landmark | Stage 3 only (and string table) | Just a string change, optionally a tag rename |
| Changed SVG visual style | Stage 3 only | Renderer config |
| Added a new POI category | Stage 3 + UE import | New layer / new class |

`organize_roads.py` (which produces `roads_organized.json` and `roads_organized.txt`) does NOT need to re-run between iterations — its inputs are the procedural grid, which is stable. It re-runs only if the grid spec changes (e.g. an L3 boundary moves).
